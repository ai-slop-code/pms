# ADR-001: Persistence layer — stay on SQLite, revisit Postgres later

- **Status:** Accepted (spike only; not a deliverable this cycle).
- **Date:** 2026-04-24.
- **Deciders:** Engineering + product.
- **Supersedes:** n/a.

## Context

PMS currently uses a single SQLite database file with WAL mode, accessed through the pure-Go `modernc.org/sqlite` driver. The production review [PMS_10](../../spec/PMS_10_Production_Readiness_Review.md) flagged the single-file setup as a concentration of risk (§3.5) and the implementation plan [PMS_11](../../spec/PMS_11_Production_Readiness_Implementation_Plan.md) tracked a Postgres option as **T3.6 — spike only**.

The questions this ADR answers:

1. Should we migrate to Postgres now?
2. If not now, under what conditions should we revisit?
3. What does the migration surface look like if we do?

## Decision

**Keep SQLite for the foreseeable future.** Revisit Postgres when at least one of the following triggers fires:

- **T-multi-writer:** a second PMS instance needs write access to the same database (e.g. active-active HA, blue/green deploys sharing state, a worker process).
- **T-concurrency:** sustained write contention causes `SQLITE_BUSY` retries above a handful per day on the deployed instance.
- **T-row-count:** any table exceeds ~10 M rows and query latency degrades beyond p95 targets.
- **T-ops:** the operator cannot tolerate the current backup posture (hourly `VACUUM INTO` + operator-managed off-host rsync, see [backup docs](../deployment/README.md)).

None of these hold today. The product is a single-tenant PMS instance with at most a few hundred properties and a few hundred thousand rows across all tables. All schedulers run in-process. Litestream is a viable add-on if continuous WAL replication is desired before any of the triggers fire.

## Consequences of staying on SQLite

Pros:

- Zero operational dependencies. `pms-server` plus a disk volume is the entire stateful surface.
- Backups are trivial: `VACUUM INTO` produces a single self-contained file.
- Latency stays low because every query is in-process.
- `modernc.org/sqlite` is CGo-free, so the release binary is fully static.
- Migrations are small, ordered SQL files we already ship in `backend/internal/migrate/`.

Cons:

- One writer at a time (SQLite locks the whole database briefly during writes).
- No native replication; Litestream or application-level snapshot rotation is the only option.
- `JSON1` and `window functions` are available, but recursive CTEs and `MATERIALIZED` views are limited vs. Postgres.
- Some operations (bulk imports >10 k rows) spike memory.

## What the migration surface looks like (for future reference)

If any trigger fires and we do choose to migrate, the work splits into three buckets.

### 1. Driver swap

- Replace `modernc.org/sqlite` imports with `github.com/jackc/pgx/v5/stdlib`.
- The `*sql.DB` abstraction means most code does not need to change.
- `dbconn.Open(path)` grows a variant that accepts a DSN (`postgres://user:pw@host/db?sslmode=require`).
- Remove the `PRAGMA journal_mode=WAL;` bootstrap and the SQLite-specific `PRAGMA foreign_keys=ON;`.

### 2. Dialect-level SQL edits

Scan of [`backend/internal/store`](../../backend/internal/store) and [`backend/internal/migrate`](../../backend/internal/migrate) shows the following incompatibilities that must be rewritten:

| SQLite usage | Postgres replacement |
|---|---|
| `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGSERIAL PRIMARY KEY` or `GENERATED BY DEFAULT AS IDENTITY` |
| `datetime('now')`, `datetime('now', ?)` | `now()` and `now() + interval '…'`; parameters need explicit casts |
| `strftime('%Y-%m', col)` | `to_char(col, 'YYYY-MM')` |
| `COLLATE NOCASE` indices | `CITEXT` column type or `lower(col)` functional indices |
| `BOOLEAN` stored as 0/1 | Native `BOOLEAN` |
| `REAL` for money | **DO NOT CARRY OVER** — switch to `NUMERIC(18,2)` during migration |
| `ON CONFLICT(...) DO UPDATE` (mostly identical, verify) | Same syntax; verify partial-index conflict targets |
| `REPLACE INTO` | `INSERT ... ON CONFLICT ... DO UPDATE` |
| Text concat via `||` | Identical |
| Unquoted identifiers are case-folded to lower | Quoted identifiers preserve case — audit every quoted table/column |

### 3. Operational changes

- Add Postgres service to `deploy/docker-compose.yml` or consume an external managed DB.
- Replace in-app backup scheduler with `pg_dump` to off-host storage (or rely on managed-DB snapshots).
- Update [`docs/deployment/README.md`](../deployment/README.md) — new env var `DATABASE_URL`, new restore runbook.
- Run the existing migrations against a staging Postgres first; the migration engine (`backend/internal/migrate`) is already dialect-aware via SQL files, so we would split `000001_init.up.sql` → `000001_init.up.sqlite.sql` and `000001_init.up.postgres.sql`.
- Convert the secret-encryption backfill path (`PMS_ENCRYPT_BACKFILL=true`) to a one-shot data migration rather than a startup scan — Postgres doesn't need the same boot-time re-encryption dance because `VACUUM INTO` is not in play.

### Data migration

For a one-time cutover:

1. Stop the PMS backend.
2. Run an export from SQLite: `sqlite3 pms.db ".dump" > dump.sql`.
3. Hand-translate DDL differences (see table above); `pgloader` can automate most of this.
4. Re-encrypt money columns into `NUMERIC` during the load.
5. Run a read-only verification pass comparing row counts and checksums by table.
6. Point PMS at the new DSN, run a smoke test, open traffic.

Downtime target: under 30 minutes for the current data volume.

## Estimated effort (if triggered)

- Driver swap + test suite stabilisation: 2–3 days.
- SQL dialect rewrite + migration splits: 3–5 days.
- Data migration + verification tooling: 1–2 days.
- Operational changes + docs: 1 day.
- **Total: about 2 weeks of focused work**, assuming no new product features are demanded in parallel.

## Decision record

We acknowledge that Postgres is the obvious destination if the app grows beyond single-tenant single-writer. Until a trigger fires, the operational and packaging simplicity of SQLite outweighs the benefits of migrating early. This ADR is the source of truth for reopening the question; any engineer may reopen it by attaching evidence that a trigger has fired.
