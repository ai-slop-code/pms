# PMS_19 - Bulletproof Booking.com ICS Reconciliation Spec

> Audience: product / property manager + implementing engineer.
> Scope: Booking.com iCal/ICS occupancy import, row identity, split-night handling, cancellation/deletion reconciliation, and duplicate prevention.
> Status: future feature specification only; no implementation in this change.

## 1. Problem Framing

Booking.com's public iCal feed is an availability feed, not a reservation API. PMS must treat every Booking.com ICS event as an **unavailable block** first. The event tells PMS that Booking.com is not selling those nights, but it does not reliably identify whether the block is one real guest stay, multiple back-to-back one-night guest stays, a manual Booking.com closure, or an externally sold block.

The current PMS importer is mostly idempotent for a stable upstream `UID`, but it does not have a complete identity model once PMS creates named stays or split rows from that upstream event. PMS can create or retain several active rows that all represent the same Booking.com unavailable night.

This breaks trust in the occupancy calendar:

- A day can show `2 nights` even though there is only one apartment and one real occupied night.
- A day detail can show both the current upstream Booking.com block and a stale/named row from an earlier sync.
- A canceled or removed Booking.com stay can remain visible when its upstream event disappeared from the feed.
- A multi-night Booking.com block can contain multiple real one-night guests, but PMS currently has no first-class way to name only one selected night/range without risking duplicate occupancy.

This is a bug in reconciliation, not only a UI wording issue. The UI is showing rows that the sync layer allowed to coexist.

## 2. Incident From July 2026

The raw feed contains this upstream Booking.com event:

```ics
BEGIN:VEVENT
DTSTART;VALUE=DATE:20260709
DTEND;VALUE=DATE:20260712
UID:3335c09bc362baaf67844f69ece8c3f4@booking.com
SUMMARY:CLOSED - Not available
END:VEVENT
```

Per RFC 5545, `DTEND` is exclusive for all-day events. This event covers exactly these nights:

- `2026-07-09 -> 2026-07-10`
- `2026-07-10 -> 2026-07-11`
- `2026-07-11 -> 2026-07-12`

PMS must never display two normal active stays for any of those nights unless the property explicitly supports multiple units. For this PMS property, capacity is one apartment, so two overlapping active rows are a data conflict.

The screenshot shows `2026-07-11` containing both:

- `2026-07-09 -> 2026-07-12 - CLOSED - Not available`
- `2026-07-11 -> 2026-07-12 - Koilpitchai`

That is not a valid final state. The named one-night row is valuable operator-entered information and should be the active stay for that selected night, but it must not coexist with the aggregate upstream row as another active occupancy for the same night.

The feed still contains the `2026-07-30 -> 2026-07-31` night, so PMS may keep July 30 blocked. The canceled stay was the following `2026-07-31 -> 2026-08-01` night, which is not present in the feed and must not remain active after a successful full sync. If any duplicate July 30 rows exist, PMS must still collapse them into one active representation for that upstream/night.

## 3. Current Implementation Risks To Fix

The current implementation has these known risk points:

- `occupancies` uses `(property_id, source_event_uid)` as the only upstream identity. That works for one row per ICS `UID`, but generated split rows use synthetic UIDs such as `UID#night-YYYYMMDD`, and manual split rows use `manual_split:UID:...`.
- The sync deletion pass keeps only the exact UIDs generated by the current run. It does not have a durable parent/child relationship between the upstream Booking.com `UID` and all rows PMS derived from it.
- Auto-splitting of `CLOSED - Not available` events depends on historical Nuki state. The same upstream `UID` can therefore be represented as an aggregate row in one run and as multiple generated night rows in another run.
- Manual split rows use `source_type = 'manual'`, so source deletion reconciliation for `source_type = 'booking_ics'` cannot see or expire them.
- Raw events are saved after expansion, which can blur the audit trail between the upstream ICS component and PMS-generated representation rows.
- A partial parse can skip events. If absence reconciliation runs after a partial parse, PMS can incorrectly mark valid source rows as deleted.

## 4. Definitions

- **Unavailable block**: one date range from Booking.com ICS where Booking.com says the property is unavailable. This is not automatically a guest stay.
- **Upstream event**: one `VEVENT` from the Booking.com ICS feed, identified primarily by the ICS `UID`.
- **Upstream UID**: the exact trimmed ICS `UID` from Booking.com, for example `3335c09bc362baaf67844f69ece8c3f4@booking.com`.
- **Source event snapshot**: the parsed upstream event for a single sync run before PMS expands or suppresses it.
- **Named stay**: the operator-selected guest stay range inside an unavailable block. It has a guest/display name and is eligible for Nuki code generation.
- **Unnamed block night**: an unavailable night from Booking.com that has not been converted into a named stay. It blocks occupancy but should not be treated as a real guest for Nuki/message flows.
- **Representation row**: an `occupancies` row shown to PMS users and consumed by downstream modules.
- **Aggregate representation**: one representation row that spans a full multi-night upstream event (its per-night coverage lives in `occupancy_nights`, section 6.1.1). "Aggregate" describes shape, not a distinct `representation_kind`; in the new model an aggregate is an `unnamed_block` row for the whole range, or a legacy row awaiting repair.
- **Manual split representation**: one PMS-created row derived from an upstream event because the operator selected a stay range or labelled one night/range.
- **Current feed**: the complete set of successfully parsed upstream events from one successful sync run.
- **Disappeared event**: an upstream UID that previously existed for this property/source but is absent from the current feed.
- **Superseded row**: a representation row that is no longer the active way PMS represents its upstream event/night but is retained for audit/history.
- **Property-local night**: a `YYYY-MM-DD` date in `properties.timezone`. For all-day Booking.com `VALUE=DATE` events, the date labels from the feed become property-local night labels; storage may use UTC midnights, but comparison and uniqueness must use property-local dates.
- **Active occupancy night**: one property-local night covered by exactly one active, non-superseded representation row. For the current PMS property, capacity is one, so this count must never exceed one.

## 5. Core Product Decisions

- Booking.com ICS is the source of availability truth for Booking-imported rows, but a Booking.com ICS event is an unavailable block, not automatically a guest stay.
- The operator creates or names actual guest stays inside unavailable blocks. When opening a blocked night, PMS must offer a stay-name/date-range flow.
- The stay-name/date-range flow defaults to exactly one night: the clicked property-local night through the next morning.
- The operator may extend the stay range in the UI, but only inside the latest Booking.com unavailable block range and only where no other active stay already covers the selected nights.
- Nuki code generation requires a named stay. PMS should not generate a guest code from an unnamed Booking.com block.
- Cleaning coordination is time-sensitive. Unnamed Booking.com block nights must create provisional cleaning events by default, using the one-night assumption, until the operator names/edits the stay range or marks the block as closed/no-guest.
- PMS must store upstream identity separately from row identity.
- PMS must reconcile at upstream-event scope first, then at property-local night scope.
- A successful full sync must be authoritative for current/future Booking.com source events.
- Failed or partial syncs must not mutate occupancies at all.
- Today/future source rows that disappear from Booking.com must become `deleted_from_source`, not remain active, even if the operator previously named the stay.
- Historical rows should be retained for audit unless an explicit repair command is requested.
- Manual/named stay rows must be reconciled strictly to the latest Booking.com date range. If Booking.com removes or shrinks the block, out-of-range today/future named rows become `deleted_from_source`.
- PMS must not fuzzy-merge a new Booking.com UID with an old Booking.com UID by date overlap. A changed UID is treated as a new unavailable block, and the old UID is reconciled as disappeared if it is absent from the latest successful full feed.
- For a capacity-one property, sync/repair must resolve overlapping active occupancies. The normal UI must not show a conflict as ordinary `2 nights` occupancy.

## 5.1 Named Stay Workflow

The Occupancy UI must support this workflow from a blocked calendar night:

1. User clicks a Booking.com unavailable night.
2. PMS opens the day detail and shows whether the night is unnamed or already assigned to a named stay.
3. If unnamed, PMS offers `Name stay / create guest stay`.
4. The form defaults to `check-in = clicked night`, `check-out = clicked night + 1 day`.
5. The form lets the user change check-in/check-out, but the selectable range is limited to nights covered by the latest Booking.com unavailable block and not already assigned to another active stay.
6. The name is required before Nuki code generation.
7. Saving creates or updates a named stay representation for the selected range.
8. The selected nights stop being represented by the generic unnamed block in active occupancy counts.

Example: Booking.com publishes one block `2026-07-09 -> 2026-07-12`. PMS may initially show three unnamed blocked nights. If the user names only `2026-07-11 -> 2026-07-12` as `Koilpitchai`, then July 11 is represented by that named stay, while July 9 and July 10 remain unnamed blocked nights. PMS must not also count the original aggregate block over July 11.

## 5.2 Generated Split Clarification

This spec intentionally separates **manual/named stays** from **legacy generated split rows**.

Example of a manual/named stay:

- Booking.com sends one block `2026-07-09 -> 2026-07-12`.
- User clicks July 11, enters `Koilpitchai`, keeps the default one-night range `2026-07-11 -> 2026-07-12`, and generates a Nuki code.
- This is a real product action. PMS should keep this named stay as long as Booking.com still blocks July 11.

Example of a legacy generated split row:

- Old PMS code sees one Booking.com block `2026-07-09 -> 2026-07-12`.
- Because a previous Nuki code existed only for one night, PMS automatically creates hidden rows like `UID#night-20260709`, `UID#night-20260710`, `UID#night-20260711`.
- The user did not explicitly create these rows as separate stays. They are an implementation artifact.

Decision: new sync code must not create hidden generated guest stays. It may materialize unnamed block-night coverage internally, but actual guest stays are created by the operator through the named-stay workflow. Existing legacy generated rows must be repaired into either named stays or unnamed block coverage.

## 5.3 No-Guest / Closure Workflow

Not every Booking.com unavailable block is a real guest. Owners close nights for maintenance, personal use, or price protection. For those nights PMS must stop creating provisional cleaning events and must exclude the night from guest occupancy.

The Occupancy UI must support a `Mark as closed / no guest` action:

- Available from a blocked night day-detail, on `unnamed_block` nights and on `named_stay` rows.
- Also available on multi-night selections, same range rules as the named-stay flow: defaults to one night, editable, limited to the latest Booking.com block range.
- On an already-named stay, the action asks for confirmation because it removes the guest name from active use (the name is preserved for audit).

Effect of marking closed / no guest:

- The affected nights get `representation_kind` context of a closure and `closure_state = 'closed'` (reuse PMS_14; see section 5.5).
- No Nuki code, no guest messages, no provisional or named cleaning event.
- Any existing provisional or named cleaning event for those checkout dates is removed immediately.
- Excluded from guest occupancy and from bookable/occupied analytics per PMS_14.
- The night still shows on the calendar as closed/unavailable so the operator knows Booking.com is blocking it.

Reversing the action:

- The operator may clear the closure to return the night to unnamed block coverage (which re-enables provisional cleaning) or to a named stay.
- Reversal follows the same immediate reconciliation and audit rules as applying it.

External sale is a distinct label from closure and keeps its PMS_14 meaning: the night is a real sold night from another channel, counts as occupied, and keeps cleaning. Marking external sale is a separate action, not part of the no-guest flow.

## 5.4 Cleaning Event Identity And Provisional Event Visibility

Provisional and named cleaning events must be idempotent so sync never creates duplicate Google Calendar events.

Cleaning event identity key:

- `property_id`
- `upstream_event_uid`
- `checkout_date` (property-local `YYYY-MM-DD`)
- `cleaning_kind` (`provisional_block` or `named_stay`)

Rules:

- One cleaning event per identity key. Re-running sync must upsert by key, never insert a duplicate.
- When an unnamed block night becomes part of a named stay, PMS removes the `provisional_block` events for the newly-covered checkouts and creates/keeps the `named_stay` checkout event by its own key.
- When a named stay is deleted, shrunk, or marked no-guest, PMS removes the corresponding `named_stay` key event.
- When a source event disappears, PMS removes all future events for that `upstream_event_uid` regardless of `cleaning_kind`.
- Removal must use the identity key so PMS never deletes an unrelated event.

Visibility (operator clarity):

- The cleaning lady only sees the Google Calendar; she does not see PMS internal state. Therefore the Google Calendar event title/description for a provisional-block cleaning should read as a normal cleaning task, not expose PMS-internal wording.
- The PMS operator, however, needs to distinguish provisional from confirmed cleaning. In PMS Occupancy and Cleaning views:
  - Show a `Provisional cleaning (from Booking block)` badge on cleaning events created from `unnamed_block` nights.
  - Show a normal cleaning indicator once the night is part of a `named_stay`.
  - When one Booking block produces several provisional checkouts (for example July 10, 11, 12), the day details must make clear these are provisional one-night assumptions that collapse once the operator names the real stay.

## 5.5 representation_kind vs closure_state

These two fields describe different things and must not duplicate meaning:

- `representation_kind` describes the row's shape/origin: `unnamed_block`, `named_stay`, `manual_closure`, `external_sale`, `synthetic_finance`, `legacy_generated_night`.
- `closure_state` (existing PMS_14 column) describes the business label: `closed`, `external_sale`, or `NULL`.

Contract:

- The no-guest/closure action sets `closure_state = 'closed'`. It does not invent a new closed concept.
- The external-sale action sets `closure_state = 'external_sale'` per PMS_14.
- `representation_kind` values `manual_closure` and `external_sale` are optional convenience shapes; if used, they must stay consistent with `closure_state`. When in doubt, `closure_state` is the source of truth for analytics, Nuki, cleaning, and finance behavior, exactly as PMS_14/PMS_17 already define.
- Downstream modules must key business decisions off the established labels (`status`, `closure_state`, `stay_outcome`, cleaning-calendar exclusion) plus the new active/superseded rules, not off `representation_kind` alone.

## 5.6 Row State Eligibility Matrix

This matrix is the implementation contract for downstream modules. A row can be preserved for audit while still being ineligible for operational workflows.

| Row state | Calendar default | Cleaning calendar | Nuki code | Guest messages | Invoice candidates | Analytics occupancy |
|---|---:|---:|---:|---:|---:|---:|
| `unnamed_block`, active, not superseded | Show as unavailable block | Provisional yes | No | No | No by default | Counts as blocked/occupied |
| `named_stay`, active, not superseded | Show as named stay | Yes | Yes | Yes | Later phase / normal stay candidate | Counts as occupied |
| `manual_closure`, active, not superseded | Show as closed | No by default | No | No | No | Excluded from bookable/occupied nights per PMS_14 |
| `external_sale`, active, not superseded | Show as external sale | Yes | No unless future channel support says otherwise | No by default | Later phase / external sale candidate | Counts as occupied |
| `deleted_from_source` | Hidden by default; visible in deleted filter | Remove future event | Revoke future code | No | No active candidate | Excluded |
| `cancelled` | Hidden by default; visible in cancelled/deleted filter | Remove future event | Revoke future code | No | No active candidate | Excluded |
| any row with `superseded_at IS NOT NULL` | Hidden by default; visible only in audit/repair view | No | No | No | No | Excluded |

Rules behind the matrix:

- Unnamed block cleaning is intentionally optimistic because cleaning coordination is operationally urgent. PMS should create provisional cleaning work as soon as Booking.com blocks a night.
- The provisional default is one checkout per blocked night. A block `2026-07-09 -> 2026-07-12` creates provisional checkouts on July 10, July 11, and July 12 until the operator selects a different named stay range.
- When the operator names or edits a stay range, PMS reconciles cleaning events to the selected range. Example: if July 9-12 becomes one named three-night stay, PMS removes the provisional July 10 and July 11 cleaning events and keeps only the July 12 checkout event.
- If a source event disappears or a named stay moves out of the latest source range before checkout, PMS removes the future cleaning event and revokes future Nuki access where applicable.
- Invoice and finance behavior is intentionally conservative in this spec. Unnamed blocks do not become invoice candidates by default. Named stays and external sales can become candidates in a later finance-focused phase.

## 6. Data Model Requirements

### 6.1 Minimal Columns

Add durable upstream ownership fields to `occupancies` or an equivalent companion table:

- `upstream_source_type`: source namespace, initially `booking_ics`.
- `upstream_event_uid`: exact Booking.com ICS `UID` that owns this row. For rows not derived from ICS, this is `NULL`.
- `representation_kind`: one of `unnamed_block`, `named_stay`, `manual_closure`, `external_sale`, `synthetic_finance`, `legacy_generated_night`.
- `representation_date`: property-local night date for one-night rows, `NULL` for multi-night named stays or legacy aggregate rows.
- `covered_nights`: the property-local nights covered by this row. This is not a JSON column; it is materialized in the `occupancy_nights` companion table defined in section 6.1.1 so per-night uniqueness is enforceable. Do not add a second parallel mechanism.
- `superseded_at`: timestamp when PMS stopped using this row as active representation.
- `superseded_reason`: short machine-readable reason, for example `replaced_by_manual_split`, `replaced_by_aggregate`, `source_deleted`, `split_strategy_changed`.

If modifying `occupancies` directly is too invasive, create a companion table with `occupancy_id` and these fields. The implementation must still enforce the same invariants.

### 6.1.1 Night Coverage Table (Per-Night Uniqueness)

A multi-night `named_stay` is one row but covers several property-local nights. To enforce "at most one active representation per night" reliably, add a per-night coverage table rather than relying on range comparisons in application code.

Table `occupancy_nights`:

- `id`
- `property_id`
- `occupancy_id` (FK to `occupancies`, `ON DELETE CASCADE`)
- `local_night_date` (property-local `YYYY-MM-DD`)
- `upstream_source_type`
- `upstream_event_uid` (nullable for non-ICS rows)
- `active` (1 while the owning row is active/non-superseded, 0 otherwise)

Constraints and rules:

- Partial unique index: `UNIQUE (property_id, local_night_date) WHERE active = 1`. This is the hard guarantee that a capacity-one property never has two active rows on the same night.
- When a representation row becomes `deleted_from_source`, `cancelled`, or superseded, set its coverage rows to `active = 0` in the same transaction.
- When a named stay is created/edited, insert its nights and flip any conflicting unnamed-block coverage to inactive atomically.
- Reconciliation, duplicate resolution, calendar counts, and analytics should read from `occupancy_nights` for night-level truth instead of re-deriving ranges each time.
- If the property later becomes multi-unit, this table is where a `unit_id` dimension would be added; the current partial unique index encodes the capacity-one assumption explicitly.

### 6.2 Upstream Event Snapshot Table

Add or extend a raw source table that stores upstream events before expansion:

- `property_id`
- `sync_run_id`
- `source_type`
- `upstream_event_uid`
- `raw_component`
- `dtstamp`
- `summary`
- `event_start`
- `event_end`
- `sequence_num`
- `ical_status`
- `content_hash`
- `created_at`

Uniqueness must be `(property_id, sync_run_id, source_type, upstream_event_uid)`.

Generated per-night rows must not replace the upstream raw component in this table. If generated rows need their own debug trail, store that separately as derived representation metadata.

### 6.3 Uniqueness Constraints

PMS must enforce these logical uniqueness rules:

- At most one active non-superseded representation per `(property_id, local_night_date)` for the current capacity-one property.
- At most one active non-superseded representation per `(property_id, upstream_source_type, upstream_event_uid, representation_date)` when `representation_date` is not `NULL`.
- At most one active non-superseded aggregate representation per `(property_id, upstream_source_type, upstream_event_uid)`.
- A given upstream event cannot have both an active aggregate representation and active per-night/named representations for the same local night.
- A row with status `deleted_from_source` or non-NULL `superseded_at` is never active for uniqueness or occupancy-count purposes.

SQLite partial indexes can enforce much of this after the new columns exist. If a full exclusion constraint is not practical in SQLite, enforce it transactionally in the store layer and cover it with tests.

`legacy_generated_night` exists only to repair rows already created by old code paths such as `UID#night-YYYYMMDD`. New product behavior should not create hidden generated guest stays. It should create either unnamed block nights or explicit named stays selected by the operator.

## 7. Sync Algorithm Requirements

### 7.1 Successful Full Sync Pipeline

A successful sync must run in this order:

1. Acquire a property-scoped sync lease/transaction guard so two syncs cannot reconcile the same property concurrently.
2. Fetch the ICS feed with timeout, size limit, and HTTP status validation.
3. Parse all `VEVENT`s into upstream events. All-day `VALUE=DATE` events must preserve the feed's civil date labels and exclusive `DTEND`; storage may normalize to UTC midnight, but business comparisons must use property-local night labels.
4. If calendar parsing fails, finish the run as `failure` and do not change occupancies.
5. If any event-level parse errors occur, finish the run as `partial_no_mutation`; store the error details, but do not upsert, delete, supersede, or otherwise mutate occupancies.
6. Store raw upstream event snapshots before representation expansion.
7. Build the desired representation set for each upstream event.
8. Reconcile each upstream event and all of its existing child rows in one transaction.
9. Mark disappeared upstream events as `deleted_from_source` only after the full feed was parsed successfully.
10. Trigger downstream reconciliation for Nuki, cleaning calendar, messages, and analytics freshness for rows that changed status/date/kind.
11. Finish the run with counts for upstream events seen, representation rows inserted, updated, superseded, restored, and deleted-from-source.

### 7.2 Failure And Partial Sync Safety

- HTTP failures, non-2xx responses, body read failures, and calendar-level parse failures must not mutate existing occupancies.
- Event-level parse failures must not mutate occupancies. A skipped event can look absent, and partial mutation can create inconsistent date coverage.
- Sync run status must make this visible to the operator: use `partial_no_mutation`, not a generic `partial`.

### 7.3 Desired Representation Selection

For each upstream Booking.com event, PMS must build the active representation set by property-local night:

- **Named stay wins for its selected nights**: if the operator named `2026-07-11 -> 2026-07-12`, that named stay is the active representation for July 11 as long as the latest Booking.com block still covers July 11.
- **Unnamed block covers remaining source nights**: source-covered nights that are not assigned to named stays remain active as unnamed blocked nights.
- **Out-of-range named stays are deleted from source**: if a named stay's night is no longer covered by the latest successful Booking.com block for its upstream UID, that today/future night becomes `deleted_from_source`.
- **Legacy generated nights are migration-only**: old `UID#night-YYYYMMDD` rows should be converted into named stays if they have a user-entered name, or unnamed block nights if they do not. New sync code must not create hidden generated guest stays.

The chosen strategy must be deterministic. The same feed and same database state must produce the same representation set on every run.

### 7.4 Named Range Reconciliation

If the operator creates or edits a named stay range:

- The selected range must be inside the latest successful Booking.com block for that upstream UID.
- The selected range must not overlap another active named stay, closure, or external sale for the same property-local night.
- Saving the range must atomically activate the named stay and remove unnamed block coverage from the same nights.
- If the range is shortened, removed today/future nights return to unnamed block coverage only if the upstream Booking.com block still covers those nights; otherwise they become `deleted_from_source`.
- Preserve operator-entered `guest_display_name` by matching existing rows by upstream UID and local night date.
- Move or relink downstream artifacts when possible; otherwise mark old artifacts for revocation/removal.

If the latest Booking.com source range changes:

- Named stay nights that remain inside the latest source range remain active.
- Named stay nights outside the latest source range become `deleted_from_source` for today/future dates, even if they have a guest name.
- Unnamed block coverage is regenerated for source-covered nights not assigned to a named stay.
- PMS must not preserve an active named stay outside the source range just because the user named it earlier.

### 7.5 Upstream Disappearance

When a full successful feed no longer contains an upstream UID:

- Mark all current/future active representation rows owned by that upstream UID as `deleted_from_source`.
- Include unnamed block rows, named stay rows, legacy generated night rows, and manual split rows derived from that upstream UID, regardless of their current `source_type`.
- Preserve `guest_display_name`, closure labels, external-sale labels, stay outcomes, cleaning-calendar exclusion flags, and audit fields for history.
- Do not create Nuki codes, cleaning events, guest messages, invoices, or active analytics entries for deleted rows.
- Revoke or remove downstream artifacts for future deleted rows where the downstream module supports removal.

Historical cutoff must be based on the property timezone. Routine sync deletion applies to rows whose checkout is today or later in the property timezone. Rows whose checkout is before today's property-local date are historical and should not be mutated by routine sync deletion.

### 7.6 Deterministic Duplicate Resolution

For a capacity-one property, sync and repair must leave at most one active representation per property-local night. If existing data already has duplicates, resolve them in this priority order:

1. Active named stay selected by the operator.
2. Active manual closure or external-sale label selected by the operator.
3. Active unnamed block from a current successful Booking.com feed event.
4. Legacy generated night row.
5. Older aggregate/source row.

If two rows have the same priority for the same night:

- Prefer the row whose upstream UID is present in the latest successful full feed.
- If both are present, prefer the row with the newest source `DTSTAMP` when available.
- If still tied, prefer the most recently updated PMS row.
- If still tied, use the lowest numeric `occupancies.id` so the result is deterministic.

All losing today/future rows must be marked superseded or `deleted_from_source` according to why they lost. The repair/sync report must list each resolved duplicate so the operator can understand what changed.

## 8. Occupancy Calendar Requirements

- The calendar must render normal occupancy from active non-superseded rows only.
- `deleted_from_source`, `cancelled`, and superseded rows must not contribute to normal day-cell counts.
- Sync and repair must resolve overlapping active rows before the calendar renders. The calendar must not show a normal `2 nights` state for a capacity-one property.
- If unresolved duplicate active rows are detected at read time, the API should return a data-quality error/warning and exclude duplicate rows from normal counts using the same deterministic winner rules as repair. The UI should show a repair-needed banner, not count both rows as occupancy.
- The day detail must show enough metadata for debugging: source type, upstream UID, representation kind, status, last sync run, and whether the row was manually split.
- The stay list should make `deleted_from_source` rows available behind a filter but not mix them into the default active stay list.

## 9. Guest Name Preservation

Operator-entered names are local PMS annotations, not source truth. Sync must preserve them for audit/history, but preservation must never keep stale rows active.

Rules:

- If a row remains the chosen representation, keep its `guest_display_name` unless the operator clears it.
- If a named stay remains inside the latest Booking.com source range, keep the name on the same selected property-local nights.
- If a named stay becomes out-of-range because Booking.com removed/shrank/disappeared the source block, keep the name on the deleted historical row but remove it from active occupancy.
- Do not copy one aggregate name across multiple nights unless the operator explicitly selected a multi-night stay range.
- If the upstream UID disappears, keep names for audit but mark the rows `deleted_from_source`.

## 10. Downstream Module Requirements

### 10.1 Nuki

- Nuki code generation must use active non-superseded named stays only.
- PMS must not generate Nuki guest codes for unnamed Booking.com block nights.
- When a row becomes `deleted_from_source` or superseded before check-in, the linked generated code must be revoked or marked for revocation.
- If a legacy generated split row is converted to a named stay and the existing code's validity window exactly matches that named stay, PMS should relink the code to the named stay. If relinking is unsafe, revoke the old code and surface the required manual action.

### 10.2 Cleaning Calendar

- Cleaning events must use active non-superseded checkout rows only.
- Active unnamed block nights create provisional cleaning events by default, one checkout per blocked night.
- Active named stays create cleaning events for the selected checkout date only.
- If a future/current row becomes deleted or superseded, its PMS-managed cleaning event must be removed.
- If naming/editing a stay range changes checkout dates, cleaning reconciliation must run immediately after occupancy reconciliation and remove obsolete provisional events.

### 10.3 Analytics

- `deleted_from_source`, `cancelled`, and superseded rows must not count as active nights.
- Duplicate active rows must never double count. Repair/sync should resolve them before analytics runs; read-time analytics must still guard against double counting by property-local night.

#### 10.3.1 Two Occupancy Metrics

An unnamed Booking.com block might be a real guest or an owner/maintenance close. Counting all blocks as guest occupancy inflates the number until the operator classifies them. PMS must therefore expose two distinct occupancy metrics instead of one ambiguous number:

- `availability_occupancy`: share of bookable nights that Booking.com marks unavailable. Includes `unnamed_block`, `named_stay`, and `external_sale` nights. Excludes `closed` nights per PMS_14. This answers "how full is my calendar".
- `guest_occupancy`: share of bookable nights with a real guest. Includes `named_stay` and `external_sale` nights only. Excludes `unnamed_block` (not yet confirmed as a guest) and `closed` nights. This answers "how many nights had a paying guest".

Rules:

- Both metrics use the same bookable-nights denominator defined in PMS_14 (available nights minus closed nights).
- `guest_occupancy <= availability_occupancy` always holds; the gap is unclassified Booking blocks.
- The Analytics UI must label these clearly and explain that the gap shrinks as the operator names stays or marks closures.
- Existing occupancy-rate consumers should migrate to `availability_occupancy` to preserve current behavior, and `guest_occupancy` is added alongside. Do not silently redefine the existing metric.

### 10.4 Finance And Invoices

- Existing finance mappings must survive row updates when the same real stay remains represented.
- If row identity changes because of a split, finance mappings should be moved only when the mapping is unambiguous by date/reference.
- If a source row disappears, keep finance history but stop offering the deleted row as an active invoice candidate by default.

## 11. Repair Requirements For Existing Data

Implementation must include a one-time repair path for databases that already contain duplicate active rows.

The repair should:

- Derive `upstream_event_uid` for existing rows from known UID patterns: raw Booking UID, `UID#night-YYYYMMDD`, and `manual_split:UID:...`.
- Group rows by property, upstream UID, and local night.
- Convert legacy generated rows with a user-entered `guest_display_name` into named stays for their matching local night.
- Convert legacy generated rows without a user-entered `guest_display_name` into unnamed block coverage if the latest successful source still covers that night.
- Mark aggregate rows as superseded for any nights covered by named stays or unnamed per-night block rows.
- Mark today/future rows as `deleted_from_source` if their upstream UID is absent from the latest successful raw source snapshot.
- Mark today/future named rows as `deleted_from_source` if their upstream UID is present but no longer covers that row's local night.
- Resolve all duplicate active rows using the deterministic priority rules from section 7.6.
- Produce a dry-run report before applying changes.
- The dry-run report must list each affected local night, winning row, losing row, reason, and whether guest names/Nuki codes/cleaning events will be relinked or revoked.
- Never hard-delete occupancy rows.

## 11A. API Contract

All endpoints are property-scoped and gated on `permissions.Occupancy` with `permissions.LevelWrite`, matching existing occupancy override actions (PMS_17/PMS_18). Shapes are indicative; align field names with existing conventions during implementation.

Named stay:

- `POST /api/properties/{id}/occupancy-blocks/{upstreamUid}/named-stays`
  - Body: `{ check_in: "YYYY-MM-DD", check_out: "YYYY-MM-DD", guest_display_name: string }`
  - Creates a named stay inside the latest Booking.com block range. Rejects (`409`) if the range falls outside the block or overlaps another active stay/closure.
- `PATCH /api/properties/{id}/occupancies/{occupancyId}/named-stay`
  - Body: `{ check_in?, check_out?, guest_display_name? }`
  - Edits range/name. Same range/overlap validation. Triggers immediate cleaning + Nuki reconciliation.
- `DELETE /api/properties/{id}/occupancies/{occupancyId}/named-stay`
  - Reverts the range to unnamed block coverage where the source still covers those nights; otherwise `deleted_from_source`.

Closure / no-guest and external sale (reuse PMS_14 endpoints where they already exist):

- `POST /api/properties/{id}/occupancies/{occupancyId}/close` - mark closed / no guest; body `{ reason?, category?, check_in?, check_out? }`.
- `POST /api/properties/{id}/occupancies/{occupancyId}/external-sale` - mark external sale; PMS_14 body.
- `POST /api/properties/{id}/occupancies/{occupancyId}/reopen` - clear the closure/external-sale label.

Repair:

- `POST /api/properties/{id}/occupancy-repair/ics-reconciliation/dry-run` - returns the resolution report, mutates nothing.
- `POST /api/properties/{id}/occupancy-repair/ics-reconciliation/apply` - applies the same plan; gated on `super_admin` or owner per repo convention; always writes an audit entry.

Response conventions:

- Follow the existing PMS_17/PMS_18 pattern: return `200` with `{ ok: false, error }` for best-effort downstream failures (e.g. Google deletion failed) rather than `500`, so the flag/state is still saved.
- Ineligible actions (e.g. naming a range outside the block) return a `409`-style conflict/ineligible error, not `500`.

## 11B. Audit Contract

Every state-changing action in this spec must write an audit entry using the existing audit-log mechanism. Minimum event types:

- `occupancy_named_stay_created`
- `occupancy_named_stay_range_changed`
- `occupancy_named_stay_deleted`
- `occupancy_block_marked_no_guest`
- `occupancy_block_reopened`
- `occupancy_ics_row_deleted_from_source`
- `occupancy_duplicate_resolved`
- `occupancy_repair_dry_run`
- `occupancy_repair_applied`
- `cleaning_provisional_created`
- `cleaning_provisional_removed`

Each entry must capture actor (user id or `system` for sync), property id, occupancy id / upstream UID, affected local night(s), before/after state summary, and the machine reason (for example `source_deleted`, `replaced_by_named_stay`, `range_shrunk`). Sync- and repair-driven changes must be attributable to the sync run id so the operator can trace exactly which run changed their calendar.

## 12. Observability Requirements

Sync runs must expose these counts:

- `upstream_events_seen`
- `upstream_events_parsed`
- `parse_errors`
- `representations_inserted`
- `representations_updated`
- `representations_unchanged`
- `representations_superseded`
- `representations_deleted_from_source`
- `duplicate_nights_resolved`
- `legacy_generated_rows_converted`
- `named_stays_deleted_from_source`

## 13. Acceptance Tests

### 13.1 July 9-12 Multi-Night Event

Given this feed:

```ics
BEGIN:VEVENT
DTSTART;VALUE=DATE:20260709
DTEND;VALUE=DATE:20260712
UID:3335c09bc362baaf67844f69ece8c3f4@booking.com
SUMMARY:CLOSED - Not available
END:VEVENT
```

When PMS syncs successfully, then each local night from July 9 through July 11 must have exactly one active non-superseded representation.

If no named stays exist yet, the nights may be represented as unnamed blocked nights. PMS must not treat the three-night block as one guest stay that automatically receives a Nuki code.

### 13.2 Named One-Night Row Must Not Duplicate Aggregate

Given a named row `2026-07-11 -> 2026-07-12 - Koilpitchai` derived from upstream UID `3335c09bc362baaf67844f69ece8c3f4@booking.com`, when the next feed still contains the July 9-12 upstream event, then PMS must keep the named row as the active July 11 representation and use unnamed blocked-night representation for July 9 and July 10.

The occupancy calendar must not show `2 nights` for July 11.

### 13.3 Canceled July 31 Event Disappears

Given an active named future row for `2026-07-31 -> 2026-08-01` owned by upstream UID `old-july31@booking.com`, when a full successful feed no longer contains `old-july31@booking.com`, then PMS must mark all active future/current representation rows owned by that UID as `deleted_from_source`, even though the row has a guest name.

The row must disappear from the default occupancy calendar and active stay list while remaining available in the deleted filter for audit.

### 13.4 July 30 Event Still Present

Given the feed still contains:

```ics
BEGIN:VEVENT
DTSTART;VALUE=DATE:20260730
DTEND;VALUE=DATE:20260731
UID:be776189886f75a948567e0d2bbe80b9@booking.com
SUMMARY:CLOSED - Not available
END:VEVENT
```

Then PMS must keep exactly one active representation for `2026-07-30 -> 2026-07-31` owned by that UID. If older rows for the same night exist under a different disappeared UID, those older rows must be marked `deleted_from_source` after a full successful sync.

### 13.5 Partial Parse Does Not Delete

Given a feed where one event parses and another event fails event-level parsing, when sync finishes as `partial_no_mutation`, then PMS must not insert, update, supersede, or mark any occupancy as `deleted_from_source`.

### 13.6 Idempotency

Given the same feed and database state, running sync twice must not change the set of active representation rows on the second run. Only sync-run metadata and `last_synced_at` may change.

### 13.7 Latest Source Range Shrinks Named Stay

Given upstream UID `abc@booking.com` previously covered `2026-07-09 -> 2026-07-12`, and PMS has a named stay for `2026-07-11 -> 2026-07-12`, when the latest successful feed changes `abc@booking.com` to `2026-07-09 -> 2026-07-11`, then the July 11 named stay must become `deleted_from_source` for today/future dates.

The guest name remains stored for audit, but the row must disappear from the default calendar and must not generate Nuki, cleaning, messages, invoices, or active analytics.

### 13.8 UID Change Creates New Block

Given old upstream UID `old@booking.com` covered `2026-08-07 -> 2026-08-08`, and the next full successful feed instead contains new upstream UID `new@booking.com` for the same date, PMS must treat `new@booking.com` as a new unavailable block and mark today/future rows owned by `old@booking.com` as `deleted_from_source`.

PMS must not fuzzy-merge the old UID and new UID by date overlap.

### 13.9 Capacity-One Duplicate Resolution

Given two active non-superseded rows overlap the same local night before repair, when sync or repair runs, then exactly one active row may remain for that local night. The losing row must be superseded or `deleted_from_source`, the report must explain the decision, and analytics must not double count that night.

### 13.10 Named Stay UI Default

Given a user clicks an unnamed blocked night `2026-07-11`, when the name-stay form opens, then it defaults to check-in `2026-07-11` and check-out `2026-07-12`. The user may extend the range only within the current Booking.com block and only across unassigned nights.

### 13.11 Provisional Cleaning For Unnamed Blocks

Given Booking.com publishes `2026-07-09 -> 2026-07-12` and no named stays exist yet, PMS must create provisional cleaning events for checkouts `2026-07-10`, `2026-07-11`, and `2026-07-12`.

When the operator names the whole range as one stay `2026-07-09 -> 2026-07-12`, PMS must remove the provisional July 10 and July 11 cleaning events and keep only the July 12 checkout cleaning event.

When the upstream UID disappears before checkout, PMS must remove all future provisional cleaning events for that UID.

### 13.12 No-Guest Closure Removes Cleaning And Guest Occupancy

Given an unnamed blocked night `2026-07-15` with a provisional cleaning event for checkout `2026-07-16`, when the operator marks `2026-07-15` as closed / no guest, then PMS must remove the provisional cleaning event, generate no Nuki code, exclude the night from `guest_occupancy`, and still show the night as closed/unavailable on the calendar.

When the operator later reopens the night and the Booking.com block still covers it, PMS must restore unnamed block coverage and recreate the provisional cleaning event.

### 13.13 Per-Night Uniqueness Is Enforced By Constraint

Given an attempt to activate two representation rows covering the same property-local night for a capacity-one property, the `occupancy_nights` partial unique index must reject the second active coverage row, so the invariant holds even if application logic has a bug.

### 13.14 Cleaning Event Idempotency

Given the same feed synced twice with an unnamed block `2026-07-09 -> 2026-07-12`, PMS must not create duplicate Google Calendar cleaning events. Each `(property_id, upstream_event_uid, checkout_date, cleaning_kind)` key must map to exactly one cleaning event.

## 14. Implementation Order

1. Add upstream ownership fields and raw upstream snapshot storage.
2. Add the `occupancy_nights` per-night coverage table with the partial unique index.
3. Backfill upstream ownership and night coverage for existing rows using known UID patterns.
4. Add store helpers that list all representation rows for an upstream UID across `booking_ics`, legacy generated, named stay, and manual split rows.
5. Change sync to build desired coverage by property-local night inside one property-scoped transaction.
6. Make partial syncs no-op for occupancy mutations.
7. Reconcile disappeared upstream UIDs across all child representation rows for today/future checkouts.
8. Add deterministic duplicate resolution using section 7.6.
9. Add cleaning event identity keys and provisional cleaning creation/removal.
10. Add the named-stay/date-range UI with one-night default and range limits.
11. Add the no-guest/closure action and reversal (reuse PMS_14 endpoints/labels).
12. Add the two occupancy metrics (`availability_occupancy`, `guest_occupancy`).
13. Add the API endpoints (section 11A) and audit events (section 11B).
14. Add the dry-run and apply repair command for existing duplicate rows.
15. Add acceptance tests before enabling the repair in production.

## 15. Non-Goals

- This spec does not add Booking.com Connectivity API support.
- This spec does not try to infer the actual guest name from Booking.com ICS; the feed does not provide it.
- This spec does not solve multi-unit inventory. It assumes the current property behaves as one sellable apartment unless a future capacity model says otherwise.
- This spec does not hard-delete historical occupancy rows.

## 16. Definition Of Done

- A successful full sync cannot leave active duplicate rows for the same upstream UID/night.
- Disappeared future/current Booking.com events become `deleted_from_source` after a full successful sync.
- Partial syncs do not mutate occupancies at all.
- Operator-entered names are preserved for audit/history without causing stale active occupancy.
- A named stay can be created from a blocked night, defaults to one night, and can be extended only inside the current Booking.com block.
- Nuki code generation is available for active named stays, not unnamed Booking.com blocks.
- Unnamed active Booking.com block nights create provisional cleaning events using the one-night assumption.
- Naming/editing a stay range reconciles provisional cleaning events to the selected checkout date(s).
- Cleaning events are idempotent by their identity key and never duplicated in Google Calendar.
- The operator can mark a block/stay as closed / no guest, which removes cleaning and excludes the night from guest occupancy.
- Per-night uniqueness is enforced by the `occupancy_nights` partial unique index, not only by application logic.
- Analytics exposes both `availability_occupancy` and `guest_occupancy` without silently redefining the existing occupancy rate.
- Every state-changing action writes an audit entry attributable to a user or sync run.
- Downstream modules follow the row-state eligibility matrix in section 5.6.
- The occupancy calendar no longer renders duplicate active rows as normal `2 nights` occupancy.
- Existing data can be repaired with a dry-run report and no hard deletes.
- Tests cover the July 2026 raw ICS examples and strategy-change edge cases.
