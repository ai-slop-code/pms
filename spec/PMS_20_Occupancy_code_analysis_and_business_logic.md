Here is my vision for the business logic that we must implement and what has to be in place.
The whole concept revolves around 2 main objects:
* Raw booking date block
* Named stay

Raw booking date block is basically an object/state that PMS syncs from ICS to its internal database. It is the foundation for creating named stays. However, raw booking date block is not counted in any analytics and is not used for final PMS business logic until it becomes a named stay. Raw booking date block simply tells PMS there is something taking place at the property from where the block comes and it is automatically synced every 15 minutes, configurable in the PMS config.

Raw booking date block creates provisional Google Calendar cleaning placeholders for every day of the block, because most stays are one-night stays and the cleaning team needs early visibility. At this stage the cleaning event is generic and just says "Upratovanie". These cleaning events are not considered final business truth. Once the relevant raw booking date range is promoted to a named stay, PMS must reconcile the relevant Google Calendar dates: confirm or update the needed cleaning event, remove unnecessary provisional cleaning events for multi-night stays, and remove cleaning events when the named stay has cleaning disabled.

Named stay is the main thing, it is an object where PMS gains its knowledge about stays, maintenance and analytics. Named stays are created from raw booking date blocks by the user of PMS. In the occupancy calendar view user can click calendar cells and then they get a dialog where they name the named stay, select the check-in and check-out date. Select what type of stay it is: booking.com stay, external stay, maintenance, personal use. Named stay can span only a part of the raw booking date block, so the leftover date should be kept until the user turns it into a named stay.

For named stays of type: booking.com stay, external stay there is always a cleaning event generated by default. However PMS user has the option at any time to update the named stay and set no cleaning which triggers Google Calendar reconciliation. For maintenance, personal use there is no cleaning event generated by default, however user can mark cleaning event and then the cleaning event gets created in Google Calendar.

Named stays are used for Nuki code generation, raw booking date blocks are not even visible in the Nuki Access section.

Named stays are also mapped to payout / reservation data once the data gets uploaded.


Stay lifecycle:
```
1) Raw booking date block - automatically synced every 15min from booking.com ICS:
2) User clicks any calendar cell where raw booking date block spans the cells
3) Dialog is displayed to the user where they can select date range to promote raw booking date block to named stay. If there is left over date range of raw booking date block left, then that's not promoted to named stay, it must be promoted later.
4) User must input the Name, check-in, check-out date, Stay type.
5) Once named stay is created PMS reconciles provisional cleaning placeholders and creates, updates, or removes Google Calendar cleaning events, if applicable.
6) Once named stay is created PMS triggers Nuki Access code generation automatically, if applicable.

```

Important business rules:
1) All stay and block date ranges should use check-in inclusive and check-out exclusive semantics, same as ICS DTEND. For example, 2026-07-09 to 2026-07-12 means nights of 9th, 10th and 11th, with checkout on 12th.
2) Named stays must be the source of truth for analytics, Nuki access, payout mapping and final cleaning state.
3) Raw booking date blocks must preserve unpromoted leftover dates. If a user promotes only part of a raw block, the remaining dates stay visible as raw booking date coverage until promoted later or removed by a future ICS sync.
4) PMS should prevent overlapping named stays for the same property unless there is a deliberate business reason to allow it.
5) Google Calendar reconciliation should only modify PMS-owned cleaning events. Matching should preferably use stored Google Calendar event IDs or metadata where available; wording-based matching should be a fallback.
6) Future ICS syncs may add, update, or remove raw booking date blocks and provisional cleaning placeholders, but must not automatically delete or mutate named stays. If synced raw booking data conflicts with existing named stays, PMS should surface the conflict for user review.

Another thing to change is Google Calendar event handling. Right now we are updating every event. We should have date-scoped and idempotent reconciliation logic. PMS should only look at the date range it is currently working with. For each affected date, PMS should check what cleaning events already exist for that day, match PMS-owned cleaning events by wording/metadata, and only create, update, or delete an event when the desired state is different from the current Google Calendar state. This is needed to avoid noisy repeated calendar updates.
